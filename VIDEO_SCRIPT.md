## SECTION 2: WHAT I BUILT (2–3 min)

### Architecture Overview (30 sec)

> The system has a three-phase pipeline: Plan, Execute, Build. Here's the architecture.

**Show folder structure:**

```
main.py                 → FastAPI server, 3 endpoints
agent/
  orchestrator.py       → Coordinates the full pipeline
  planner.py            → LLM decomposes request into tasks
  executor.py           → Runs tasks in dependency order
  memory.py             → Sliding window conversation memory
  models.py             → Pydantic data models
docbuilder/
  builder.py            → Markdown to styled .docx
llm/
  client.py             → OpenRouter API wrapper
  mock_client.py        → Fallback for testing without API key
```

### Tech Stack (15 sec)

> Python, FastAPI for the API, LangChain for LLM orchestration, python-docx for document generation, Pydantic for data validation, and OpenRouter as the LLM provider using the `tencent/hy3:free` model.

### Planning Logic (45 sec)

**Show `agent/planner.py`:**

> The planner uses LangChain's LCEL chain — prompt, LLM, Pydantic parser. The LLM gets a system message asking it to break the request into 3-8 tasks with dependencies, and returns structured JSON that Pydantic validates.

```python
chain = prompt | llm | parser
plan = chain.invoke({"request": request, "format_instructions": ...})
```

> If the LLM returns invalid JSON, we retry twice, then fall back to a hardcoded 5-task template. This ensures the system never fails completely.

### Execution Logic (45 sec)

**Show `agent/executor.py`:**

> The executor runs tasks in dependency order using a DAG. `_next_ready_task` finds the first task whose dependencies are all completed. Each task gets context from two sources — results of its dependencies, and conversation memory.

```python
while True:
    task = _next_ready_task(state.tasks)
    if task is None:
        break
```

> Each task has up to 3 retries with exponential backoff. And tasks are classified as critical or non-critical — if a critical task like "Executive Summary" fails, we stop. If a non-critical task like "Supporting Data" fails, we skip it and continue.

### Document Generation (30 sec)

**Show `docbuilder/builder.py`:**

> The builder takes each task's markdown output and converts it to a styled Word document. It handles headings, bullet lists, numbered lists, bold text, and tables. Tables get a dark header row and alternating row shading using low-level XML because python-docx doesn't expose that directly.

### Engineering Improvement (30 sec)

> The mandatory engineering improvement I implemented is **retry and fallback logic** at two levels. First, the planner retries LLM calls for task decomposition and falls back to a hardcoded template. Second, the executor retries each task with exponential backoff and classifies tasks as critical vs non-critical. This means the system is resilient — it produces a document even when parts fail, rather than crashing.

## SECTION 3: DEBUGGING INSIGHT (1–2 min)

> One issue I ran into was with the document generation. When rendering tables, the code was trying to convert an `RGBColor` object to an integer using `int(color)` to create the hex string for cell shading.

**Show the error:**

```
TypeError: int() argument must be a string, a bytes-like object or a real number, not 'RGBColor'
```

> The root cause was that newer versions of python-docx changed the `RGBColor` class — it no longer supports `int()` conversion. The old code was `"%06X" % int(color)`, which worked in older versions but broke in the version I had installed.

**Show the fix:**

```python
# Before (broken)
shading_elem.set(qn("w:fill"), "%06X" % int(color))

# After (fixed)
shading_elem.set(qn("w:fill"), str(color))
```

> The fix was simple — `str(color)` on an `RGBColor` already returns the hex string like `"1A237E"`, which is exactly what the XML attribute needs. No formatting needed. This is a good example of why pinning dependency versions matters — the code worked on one machine and broke on another because of a library version difference.

## SECTION 4: TRADEOFF DISCUSSION (1–2 min)

> The main tradeoff I want to discuss is **autonomous planning vs deterministic workflows**.

> My system uses an LLM to dynamically generate the task plan for each request. The alternative would be a deterministic approach — predefined templates for different document types, like a template for proposals, one for reports, one for meeting minutes.

> The autonomous approach is more flexible. You can ask for anything and the agent figures out the steps. But it's less predictable — the LLM might create 4 tasks or 7, might include sections you don't need, or structure things differently each time.

> The deterministic approach is reliable and fast, but rigid. If someone asks for something slightly outside your templates, it either fails or produces a bad result.

> I chose autonomous planning because the assignment specifically asks for an agent that "creates its own task list" and demonstrates "decision-making." The tradeoff is acceptability — I mitigate the unpredictability with retry logic, fallback templates, and Pydantic validation to ensure the output is always structured correctly even if the LLM makes odd choices.

> The second tradeoff worth mentioning is **in-memory state vs database**. I store all agent state in a Python dict. It's simple and fast, but lost on restart and doesn't scale. For a 60-minute build challenge, this was the right call — it kept the code clean and focused on agent logic rather than infrastructure. In production, I'd use Redis or PostgreSQL.
